# 🧑‍🏫 Instructor Facilitation Guide: Module 9 – Accessibility Best Practices

> 📦 “Designing documentation that **everyone** can access, understand, and use.”

---

## 🧠 Module Overview

### 🎯 Objective
Equip learners with the knowledge and hands-on experience to write inclusive documentation that adheres to modern accessibility standards (WCAG 2.2, Section 508) and delivers equitable user experiences.

- 🧩 **Lecture & Demos:** 1.5 hours

## 📘 Module Sections

### 1. 🧊 Icebreaker & Framing (15 min)

> **Prompt:** “Describe a time you couldn’t understand or access a product. What made it difficult?”

Use real examples to emphasize **why accessibility matters** across all industries.

---

### 2. 📚 Key Concepts Breakdown (30 min)

#### 🔍 What is Accessibility?
- Inclusive design approach ensuring all users can **perceive, understand, navigate**, and **interact** with content.
- Includes people with:  
  - Visual, auditory, motor, and cognitive impairments  
  - Temporary or situational disabilities  
  - Language processing or neurological differences  

#### 🎯 WCAG 2.2 Core Principles  
> **P.O.U.R. Framework**
- **Perceivable**: Content is visible and understandable  
- **Operable**: Interface is navigable by keyboard/switches  
- **Understandable**: Language is plain and logical  
- **Robust**: Compatible with assistive tech (AT)

#### ⚖️ Legal Standards
| Regulation        | Scope                                  |
|------------------|------------------------------------------|
| WCAG 2.2 (W3C)    | International web standard               |
| Section 508 (U.S.)| Federal accessibility requirement        |
| EN 301 549        | EU procurement & digital products        |

---

### 🧑‍💻 3. Hands-On Activities (60 min)

#### 🛠️ Activity 1: Accessibility Audit  
- Audit a sample document using **WAVE** or **Axe**  
- Identify:  
  - Missing alt text  
  - Improper headings  
  - Keyboard traps  
  - Contrast issues

#### 📝 Activity 2: Accessible Rewriting  
- Transform non-compliant documentation into:  
  - Plain language  
  - Proper heading hierarchy  
  - Correct table structures  
  - Descriptive links  
  - Alt text with functional value

#### 🔤 Activity 3: Semantic HTML Drill  
> Convert a flat HTML document to fully semantic structure  
- `<header>`, `<main>`, `<nav>`, `<section>`, `<footer>`  
- Label elements for screen reader landmarks  

---

### 👥 4. Peer Review & Group Audit (45 min)

#### ✅ Group Tasks
- Pair up and review each other’s documents using:
  - **Accessibility Checklist**
  - **WCAG Tags (AA/AAA compliance)**

#### 💬 Guided Peer Feedback Prompts
- “Was this readable aloud with a screen reader?”
- “Is the language clear for ESL or cognitive challenges?”
- “Can this doc be navigated without a mouse?”

---

### 🧠 5. Empathy Simulation (Optional, 15 min)

#### 🔒 Accessibility Perspective-Taking
- Close your eyes and listen to a poorly structured document.
- Repeat with an accessible version.
- Discuss: How did you feel? What changed?

> Goal: Cultivate empathy as a **design discipline**, not charity.

---

### 🧩 6. Knowledge Check + Brain Teasers (15 min)

Use quizzes, puzzles, and discussion to drive retention.

#### 🧠 Questions
1. Name two issues a screen reader user might face in an unstructured document.
2. What’s the downside of using “Click here” in link text?
3. How can semantic HTML improve navigation for AT users?
4. What contrast ratio is considered acceptable in WCAG 2.2?
5. Describe how accessibility impacts SEO (Search Engine Optimization).
